Cost Analysis Methodology

How the Desmos Graph Analyzer v2 estimates expression cost and performance scores

These are heuristic estimates, not measured timings. The weights are tuned to surface the patterns that typically cause performance issues in Desmos graphs. The relative ordering is what matters—an expression scoring 31.0 is far more expensive than one scoring 1.5—but the numbers do not represent actual milliseconds.

⚡ Level 1: Per-Expression Cost

Every expression in the graph is assigned an estimated cost by the eCost() function. The cost starts at a base of 1 and accumulates penalties based on which computational patterns are detected in the LaTeX. Expressions over 2,000 characters (long data lists) are assigned a flat cost of 1.

Pattern DetectedPenaltyRationale
Base cost +1 Every expression has some evaluation overhead, even a simple constant like a = 5.
LaTeX length +0.003 per character Longer expressions generally involve more arithmetic operations, more function calls, or inline data. A 200-character expression adds +0.6; a 1000-character one adds +3.0.
Recursive definition +30 By far the heaviest penalty. A recursive expression like f(n) = g(f(n-1)) recomputes its entire chain from the base case on every evaluation.
Nested summation + integral +20 New in v2. When a summation wraps a definite integral, the cost is multiplicative—each term requires numerical integration. This is one of the most expensive patterns possible.
Definite integral (∫) +10 New in v2. Numerical integration is significantly more expensive than algebraic operations. Applied only when the integral is not nested inside a summation (which gets +20 instead).
List comprehension (for) +8 Generates a list by iterating, e.g., f(n) for n=[1...50]. Cost scales with list size.
Summation / total +6 Iterates over an entire list to produce a sum. Applied only when no integral is nested inside (which gets +20 instead).
repeat operation +6 New in v2. Duplicates curves or points for rendering. Each repetition requires independent GPU processing, stress-testing the rendering pipeline.
Polygon operation +4 Polygon vertex processing plus GPU rendering. Cost scales with vertex count.
with clause +2 Introduces a scoped sub-computation, adding evaluation overhead.
join operation +2 List concatenation that creates a new list in memory.
Piecewise / conditional ({...}) +1 Branching logic. Individually cheap, but many nested piecewise expressions add up.
Long data list (>2000 chars) flat 1 Hardcoded data is evaluated once and cached. The full cost formula is skipped; these are flagged but not penalized.

Example: Scoring a recursive contour function

Expression: c_ontour(n) = p_next(c_ontour(n-1))

Base cost: 1.0
LaTeX length (48 chars × 0.003): +0.14
Recursive definition: +30.0
No other patterns detected
Total: 31.1

Example: Scoring a nested sum + integral (v2)

Expression: f_hard(a) = ∑(n=1 to 10^k, ∫(0 to n, sin(t-a)dt))

Base cost: 1.0
LaTeX length (91 chars × 0.003): +0.27
Nested sum+integral: +20.0
No other patterns detected
Total: 21.3

Example: Scoring a simple constant

Expression: n_max = 50

Base cost: 1.0
LaTeX length (10 chars × 0.003): +0.03
No other patterns detected
Total: 1.0

🔗 Level 2: Per-Input Triggered Cost

For each input variable (sliders, draggable points, constants with no upstream dependencies), the analyzer computes the total cost of everything downstream—every expression that would need to recalculate when that input changes.

How it works

  1. Identify inputs — Find all variables with zero upstream dependencies (nothing feeds into them).
  2. Walk the dependency graph transitively — From each input, follow every edge in the dependedBy graph to collect all downstream expressions.
  3. Sum their costs — Add up the eCost() of every affected expression to get the total triggered cost.
  4. Rank inputs — Sort by total cost descending. The highest-cost input is the primary performance bottleneck.

Example: Variable a in the ocean current graph

Affected expressions: 28
Includes: recursive contour (31.1), trail list (9.2), polygon checks ×6 (4.0 each), images ×7 (1.0 each), ...
Sum: 31.1 + 9.2 + 24.0 + 7.0 + ... = ~142
Total triggered cost: ~142

This means every time the user moves the duck (changing a), ~142 units of computation fire. That's why dragging feels laggy.

The triggered cost directly answers the question: "Why is my graph slow when I drag X?" The higher the number, the more work happens per frame.

📊 Level 3: Performance Score (0–100)

The overall performance score provides a single at-a-glance number for the graph's health. It starts at 100 and subtracts weighted penalties based on graph-level metrics. The result is clamped to the range 0–100.

score = 100
  − recursiveDefinitions × 18   // heaviest penalty
  − nestedSumIntegrals × 15   // v2 – multiplicative cost
  − integrals × 8   // v2 – numerical integration
  − max(0, draggables + movablePoints − 1) × 5   // first is free
  − implicitCurveFamilies × 5   // v2 – list-range curve plotting
  − repeatOperations × 4   // v2 – rendering multiplier
  − max(0, images − 3) × 2.5   // first 3 are free
  − listComprehensions × 3.5
  − summations × 3
  − max(0, expressions − 50) × 0.3   // v2 – steeper when >500 exprs (was 0.15)
  − max(0, hiddenComputing − 10) × 0.4
  − polygonOps × 1.5
  − avgLatexLength × 0.015
What changed in v2? Five new factors were added (nested sum+integral, integrals, repeat operations, implicit curve families, movable points), and the expression-count penalty was steepened to −0.3 per expression beyond 50 when a graph has over 500 expressions.
FactorPenaltyFree AllowanceRationale
Recursive definitions −18 each None A single recursion can dominate frame time. Two can make a graph unusable.
Nested sum+integral v2 −15 each None When a summation wraps an integral, each iteration triggers a full numerical solve. Extremely expensive.
Integrals v2 −8 each None Numerical integration is one of the most expensive single operations Desmos performs.
Draggable / movable points v2 −5 each First 1 free v2 merges sliders, draggable elements, and movable points (constant point definitions) into a single count. Each additional one multiplies dependency chains per frame.
Implicit curve families v2 −5 each None Expressions like y = f(x, n) for n=[1...50] generate many implicit curves that each need rendering.
Repeat operations v2 −4 each None Each repeat duplicates a curve for rendering, multiplying GPU work.
Images −2.5 each First 3 free Images require GPU rendering and position computation. A few are fine; many add up.
List comprehensions −3.5 each None Each generates a list via iteration, scaling with list size.
Summations −3 each None Full-list iteration per evaluation.
Total expressions −0.3 each First 50 free v2 raised the per-expression penalty from 0.15 to 0.3 for graphs with >500 expressions, where overhead becomes significant.
Hidden computing expressions −0.4 each First 10 free Hidden folders still evaluate. Many hidden expressions waste processing silently.
Polygon operations −1.5 each None Vertex processing and rendering cost, especially with many vertices.
Average LaTeX length −0.015 per char None A proxy for overall expression complexity across the graph.

The resulting score maps to a label and color:

75–100: Good
50–74: Fair
25–49: Poor
0–24: Critical

Example: A complex graph with 1035 expressions, integrals, and repeat ops (v2 formula)

100 − 0 recursive defs × 18 = 0
− 1 nested sum+integral × 15 = −15
− 3 integrals × 8 = −24
− max(0, 4−1) draggable/movable × 5 = −15
− 2 implicit curve families × 5 = −10
− 5 repeat ops × 4 = −20
− max(0, 0−3) images × 2.5 = 0
− 2 list comprehensions × 3.5 = −7
− 1 summation × 3 = −3
− max(0, 1035−50) × 0.3 = −295.5
− max(0, 10−10) × 0.4 = 0
− 0 polygon ops × 1.5 = 0
− avg 35 chars × 0.015 = −0.53
Score: ~0 (Critical)

The massive expression count alone exceeds the 100-point budget. In v2, the steeper −0.3 per-expression penalty catches graphs that pad their count with trivial color/note expressions.

The three levels of cost estimation feed into every section of the generated report:

Severity Banner Performance score determines the banner color and label (critical / warning / good).

Root Cause Callouts Generated from the highest per-expression costs. v2 adds dedicated callouts for nested sum+integral, standalone integrals, repeat operations, implicit curve families, high expression count (>500), and movable point dependencies.

Cost Breakdown Per-expression costs summed by category. v2 adds categories for integrals (including nested in sums) and repeat operations.
Recommendations Triggered by specific metric thresholds. v2 adds recommendations for nested sum+integral, standalone integrals, repeat operations, implicit curve families, and high expression count.

🔌 Functional Dependency Graph v2

Version 2 introduces a dedicated section that maps every function and variable to its callers, callees, and the specific variables passed as arguments. This goes beyond the basic "depends-on / depended-by" graph used in v1.

How it works

  1. Parse function calls — A findFnCalls() function scans each expression's LaTeX for patterns like f(arg1, arg2), handling nested parentheses and brackets while excluding built-in functions (sin, cos, etc.).
  2. Extract argument variables — For each call site, the individual arguments are split and scanned for variable references, producing a list of variables passed to each parameter.
  3. Build call maps — Two maps are constructed:
    • callsTo[callerName] — which functions each expression calls and what variables it passes.
    • calledBy[functionName] — which expressions call this function and what variables they pass.
  4. Display in report — Functions show their signature, cost, dependencies, callers (with argument tables), and callees. Variables show their upstream/downstream connections and function calls.

Example: Function f_{low}(x, y, t)

Depends on: a, b, c_{oeff}
Called by:
  p_{ath} passes [x_{0}, y_{0}, t_{step}]
  v_{ec} passes [P.x, P.y, 0]
Calls: g_{rad}(x, y), h_{eat}(t)

This tells you that changing variable a affects f_{low}, which in turn affects p_{ath} and v_{ec}. It also shows that dragging point P triggers f_{low} via v_{ec}.

🚧 Special Handling Rules

RuleBehavior
Long data lists (>2000 chars) Assigned a flat cost of 1. Flagged in the report as informational but excluded from cost penalties. These are typically hardcoded lookup tables.
Expressions containing j_{uice} Completely skipped during analysis. These are Desmos internal/utility expressions and do not contribute to user-facing performance.
Movable points v2 Constant point definitions like P = (3, 5) are detected and counted alongside sliders as inputs. Expressions referencing P.x or P.y are tracked as dependents.
Limitations The analyzer detects patterns in the LaTeX source text using regex matching. It does not execute the expressions or measure actual runtime. Costs do not account for Desmos's internal optimizations (caching, lazy evaluation) or hardware differences. Use the scores as a guide for where to investigate, not as absolute measurements.
