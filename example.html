🔎 Root Cause Summary

Primary Bottleneck: Recursive Contour Computation The expressions c_ontour1(n) and c_ontour2(n) each recurse 50 levels deep. Every recursive step calls p_next(p), which itself calls two expensive functions: v_field(p) (bilinear interpolation in a ~700-element vector list) and v_allContinents(p) (shore-force summation over continent vertices). When you drag the duck, all 50 steps recompute from scratch on every mouse-move frame.
Secondary Bottleneck: Point-in-Polygon Checks Six separate isPointInPolygon calls run on every drag frame, each iterating over the vertices of a continent region (up to 17 vertices per region). These determine whether the duck is over land.
Rendering Load: Multiple Image Instances The rubber duck image is rendered at 7+ different positions simultaneously (wrapping copies, drag copies, animation copies, trail-head copies), each dependent on the drag point coordinates.

⚡ Expensive Helper Functions (Called per Recursive Step)

These functions are invoked inside p_next(p), which is called 50 times for each contour chain. The total cost multiplies across the recursion depth.

Critical Per-Step Function Costs

p_next(p) — Particle advection step x50 calls
95%

Combines velocity field interpolation and continental shore forces to compute the next particle position. Uses mod/min/max for wrapping.

v_field(p) — Bilinear velocity interpolation ~700 vector list
70%

Interpolates between 4 neighboring grid vectors from v_buoy (a hardcoded list of ~700 2D vectors). Calls index(p), X(p), Y(p) for grid coordinates.

v_allContinents(p) — Shore force summation N edge iterations
60%

Sums v_fromShore over every edge of the nearest continent polygon. Each edge check involves distance calculation and conditional force computation. The getClosestContinent(p) lookup adds branching cost.

v_fromShore(p1, p2, a) — Per-edge shore force per edge
35%

Calculates signed distance to a shore edge, applies cutoff threshold, and returns repulsion vector if within range. Calls distanceFromShore and distance(p1,p2).

getClosestContinent(p) — Spatial zone lookup per step
15%

Spatial partitioning: selects which continent polygon to test based on the point's position using nested piecewise conditionals (A, B, C, D, E, or F).

💡 Performance Recommendations

1

Reduce n_max (recursion depth)

The single most impactful change. Reducing n_max from 50 to 25 would halve the per-frame computation cost. The trail would be shorter but the drag interaction would be noticeably smoother. Consider whether 30–35 steps provide a good visual-to-performance tradeoff.

2

Defer contour computation to drag-end

Instead of recomputing the full 50-step contour on every mouse-move, use a CL (Computation Layer) action to capture (a, b) only on drag-end and compute the contour from the captured values. During the drag, only move the duck image without trail updates. The x_moveCaptured / y_moveCaptured variables already exist for this pattern but currently update live.

3

Eliminate hidden Contour #2

The entire "Draw contour from current position #2" folder is hidden but still computed. If it’s not needed, removing it would save a full 50-step recursive chain from the initial computation budget and free memory.

4

Consolidate point-in-polygon checks

Instead of 6 separate isPointInPolygon calls, use the existing getClosestContinent(p) spatial lookup (which is cheap) to determine which single region to test, then run only one polygon check. This would reduce the polygon test cost by ~80%.

5

Reduce duplicate duck image instances

Currently 7+ duck images render simultaneously for wrapping and animation states. Consider conditionally showing only the relevant set (drag-mode vs animation-mode) and reducing wrap-around copies if the viewport doesn’t extend near the ±180° boundary.
